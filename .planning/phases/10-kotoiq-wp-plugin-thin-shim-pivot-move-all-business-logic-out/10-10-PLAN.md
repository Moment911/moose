Build the side-by-side dual-run shadow mode per CONTEXT.md
D-TypeScript-port-equivalence (USER-LOCKED). For the first 7 days of a
site's cutover, every dashboard call to a v4 verb ALSO fires the equivalent
v3 endpoint as a shadow. Responses are diffed and logged. v4 is what the
caller sees; v3 runs read-only for comparison.

site to v4-primary (mode='promoted'). Sampled checks continue for monitoring.

sitemapCrawler.ts vs PHP originals BEFORE they affect customer sites.

           async runVerb<T>(verb: ShimVerb, args: Record<string, unknown>): Promise<ShimRpcResponse<T>> {
             if (mode === 'inactive' || mode === 'rolled_back') {
               const v3 = await callV3Endpoint(verb, args, agencyId, siteId);
               return v3.ok
                 ? { ok: true, data: v3.data as T, status: v3.status }
                 : { ok: false, error: { code: 'v3_error', message: 'v3 returned error' }, status: v3.status };
             }
             const v4Start = Date.now();
             const v4 = await shimRpc<T>(siteUrl, verb, args);
             const v4Latency = Date.now() - v4Start;
             const shouldCheckV3 = mode === 'active' || (mode === 'promoted' && Math.random() < 0.01);
             if (!shouldCheckV3) return v4;

             const v3 = await callV3Endpoint(verb, args, agencyId, siteId);
             if ((v3 as any)._no_v3) {
               await this._logDualRun(verb, null, args, null, v4.ok ? v4.data : null, 0, v4Latency, 'v4_only', null);
               return v4;
             }
             let status: string;
             let summary: any = null;
             if (!v3.ok && (!v4.ok)) status = 'both_error';
             else if (!v3.ok) status = 'v3_error';
             else if (!v4.ok) status = 'v4_error';
             else {
               const diff = compareResponses(v3.data, v4.data);
               status = diff.status;
               summary = diff.summary;
             }
             await this._logDualRun(
               verb,
               (v3 as any)._no_v3 ? null : `POST /api/wp action=${V4_TO_V3_ACTION_MAP[verb] ?? '?'}`,
               args,
               v3.ok ? v3.data : null,
               v4.ok ? v4.data : null,
               v3.latency_ms,
               v4Latency,
               status,
               summary
             );
             return v4;
           },

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dualRunRouter → /api/wp (internal fetch) | Same-origin call; uses agency-scoped service context. v3 endpoint trusts the dispatcher. |
| dualRunRouter → koto_wp_dual_run_log insert | Hashed responses only; no full bodies stored. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-10-01 | Information Disclosure | args_hash + v3/v4_response_hash leak post titles or meta values | accept | sha256 hashes are one-way; brute-forcing requires guessing pre-image which has high entropy in real-world args. Documented in Plan 01 threat model. |
| T-10-10-02 | Tampering | Operator with wrong agency sets mode='active' on another agency's site | mitigate | set_mode action does .eq('agency_id', session.agencyId) on the update; cross-agency update affects 0 rows |
| T-10-10-03 | Repudiation | A diff is logged with wrong hash because of caller bug | mitigate | hashResponse is deterministic (sorted keys); test covers determinism + null handling |
