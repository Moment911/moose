<objective>
Port four of the v3 modules to dashboard TypeScript: seo (v3 modules/seo.php
+ modules/seo-metabox.php), seo-redirects (v3 modules/seo-redirects.php),
snippets (v3 modules/snippets.php), access (v3 modules/access.php), and
search-replace (v3 modules/search-replace.php).

The dashboard now OWNS the algorithms; the v4 shim plugin delivers only
generic primitives (meta read/write, option read/write, query.select,
capability.apply, database.update_bulk).

Per research §search-replace port plan: the serialized-PHP-safe scan logic
moves to dashboard TypeScript (using `php-unserialize` npm package). The
shim's `database.update_bulk` verb (Plan 05) applies the resulting list of
{table, pk, col, value} updates. This trades some roundtrips for IP
protection — acceptable per research; benchmark called out for cutover.

- haiku-4-5 for cheap work, sonnet-4-6 for quality

Per CONTEXT.md D-Page-design-model: this plan does NOT cover page-design /
template capture flow — that's Plan 09. This plan covers porting EXISTING v3
modules' behavior to dashboard TS.

Output:
- Five port modules in `src/lib/wp-shim/ports/`
- Test coverage for each
- Re-exported via the package index for downstream UI consumers

         - Aggregate response
