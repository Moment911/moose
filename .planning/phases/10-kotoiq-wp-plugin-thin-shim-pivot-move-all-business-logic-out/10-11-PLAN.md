<interfaces>
Pairing flow end-to-end (CLI orchestration):
  1. Operator runs: node scripts/cutover/pair-site.cjs --site-id=<uuid> --agency-id=<uuid>
  2. Script prints wp-cli + WP admin instructions for opening pairing window on the site
  3. Operator opens window (either: SSH + wp-cli OR WP admin → KotoIQ Shim → Pairing → Open window)
  4. Operator confirms window is open: script polls /wp-json/kotoiq-shim/v1/health to detect plugin presence (no auth needed for unauthenticated probe is NOT exposed — instead probe via shimRpc which will return 'not_paired' if pubkey is unset; the 'not_paired' response confirms plugin is installed)
  5. Script fires POST /pair via pairSite() helper (Plan 03)
  6. Script verifies pair_completed + health_verified audit rows + confirms fingerprint matches Vercel pubkey
  7. Script sets koto_wp_sites.dual_run_state='inactive' (still using v3)

Cutover sequence per site:
  inactive → active (operator decision; triggers 7-day clock)
  active → promoted (after 7 days + 0 major_diff + manual confirm via promote-site.cjs)
  promoted → rolled_back (emergency only; via kill-switch.cjs)

Manifest endpoint update:
  Before: sha256 returns null (Plan 02 stub)
  After: sha256 reads from process.env.KOTOIQ_SHIM_DIST_SHA256 (set by build-shim-zip.sh after upload) OR a Supabase row in a new koto_dist_manifests table — research deferred which; the planner chose env var as the simpler v1.

Kill switch behavior:
  --confirm --reason='...' --mode={destruct|rollback|sample-only}
    destruct: fires shimRpc '/destruct' to every paired site (deactivates v4)
    rollback: sets every dual_run_state='rolled_back' (v3 primary; v4 inactive)
    sample-only: sets dual_run_state='promoted' for all currently 'active' sites (skips waiting)
</interfaces>

    - args: --site-id=<uuid> --agency-id=<uuid> --confirm
    - Loads site row, verifies dual_run_state='active' + dual_run_started_at >= 7 days ago
    - Queries koto_wp_dual_run_log for site in the last 7 days:
      - count major_diff entries — MUST be 0
      - count v3_error entries — MUST be 0 (v3 errors are scary)
      - count v4_error entries — MUST be 0
      - count total → MUST be > 0 (real traffic happened)
    - If all gates pass: UPDATE koto_wp_sites SET dual_run_state='promoted', v4_promoted_at=now()
    - Inserts koto_wp_shim_pairings audit row: event='promoted_to_v4'
    - --confirm flag required (refuses without explicit consent)
    - On gate failure: prints which gate failed + how to investigate

    kill-switch.cjs:
    - args: --confirm --reason='...' --mode={destruct|rollback|sample-only} --filter (optional: 'all' | 'active' | 'promoted' | site-id list)
    - destruct: for each filter-matching site, fire shimRpc('/destruct', {deactivate: true}) — deactivates v4 plugin
    - rollback: UPDATE koto_wp_sites SET dual_run_state='rolled_back' for matching sites
    - sample-only: UPDATE koto_wp_sites SET dual_run_state='promoted' (used in emergencies when traffic must shift fast)
    - Inserts koto_wp_shim_pairings rows with event='kill_switch_fired' + notes={reason, mode, filter, ts}
    - REQUIRES --confirm + --reason (refuses otherwise)
    - Prints a fleet-wide change-set BEFORE executing + asks for typed confirmation ("type 'I UNDERSTAND' to proceed"); auto-yes only with --yes flag for CI
