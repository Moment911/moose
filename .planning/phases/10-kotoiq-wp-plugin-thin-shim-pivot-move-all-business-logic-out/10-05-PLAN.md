no KotoIQ-specific information leaks.

in PHP, never accepts raw SQL, uses $wpdb->prepare with positional
placeholders, has documented limit_max.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| query.select dispatcher → wpdb | Single most security-critical surface in the plugin. SQL injection here = total site compromise. |
| capability.apply → WP_Role | Granting wrong cap to wrong role = privilege escalation across the site. |
| Snippet runtime eval() → PHP | eval is the most dangerous call in the plugin. The trust boundary is the option store: anyone who can write the option can execute arbitrary PHP. |
| webhook emitter → dashboard URL | Outbound POST — the URL must be HTTPS + dashboard-authenticated at the destination. Otherwise we leak post IDs + user IDs to attackers. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-05-01 | Tampering | SQL injection via query.select parameter | mitigate | Every named query uses $wpdb->prepare() with positional placeholders. Whitelist hardcoded in source. New named-query additions require code review + adversarial-input test per research §Pitfall 7. |
| T-10-05-02 | Tampering | A new named-query is added in a future plan with a parameter that bypasses prepare() | accept (process control) | The whitelist sits in plugin source — adding a new entry requires a PR.
