<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WP shim → dashboard receiver | Inbound webhook POST from the (locked) plugin to /api/kotoiq/wp-event — untrusted until authenticated |
| dashboard → WP shim | Outbound webhookSet RPC (already Ed25519-signed via shimRpc) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01 | Spoofing | /api/kotoiq/wp-event open receiver | mitigate | verifyWpEvent authenticates the emitter's POST (shared secret/HMAC/Ed25519 per Plan 10-05 emitter shape); reject on missing/invalid |
| T-11-02 | Tampering | webhook event payload (post_id, site_url) | mitigate | Validate event is in allowlist + site_url maps to a known koto_wp_sites row before acting (ASVS V5) |
| T-11-03 | DoS | unauthenticated flood of wp-event | accept | Auth check rejects pre-work; low-value endpoint; matches existing accepted fire-and-forget risk |
| T-11-04 | Info disclosure | cross-agency via receiver re-running snapshot | mitigate | Resolve agency_id/client_id from the authenticated site row, never from request body alone |
</threat_model>
