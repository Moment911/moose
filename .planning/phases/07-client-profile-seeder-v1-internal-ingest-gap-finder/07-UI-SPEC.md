# Phase 7 — UI Design Contract

> Visual and interaction contract for the **KotoIQ Launch Page** — the narrative ingest + gap-finder canvas that turns Koto onboarding / voice / discovery data into a living `kotoiq_client_profile`, launches `pipelineOrchestrator.ts` Stage 0, and keeps a clarification queue alive during the run.
>
> Generated by `gsd-ui-researcher`. Must be read by planner, executor, and ui-checker before any implementation task begins. This spec inherits the existing Koto design language from `src/lib/theme.ts` — it does not replace it. Every value below is prescriptive; executor chooses no colors, sizes, weights, or copy on its own.

---

## 0. Phase Overview (executor orientation)

- **Route:** `/kotoiq/launch/:clientId` — mounted as a new shell tab under `KotoIQShellPage.jsx`. Visual chrome (sidebar + top nav + shell tabs) is inherited from the shell.
- **Primary surface file (new):** `src/views/kotoiq/LaunchPage.jsx`
- **Component family (new):** `src/components/kotoiq/launch/*.jsx`
- **North-star UX beat (user's exact words):** "Free flowing, fully living breathing system. Wow the crap out of the user."
- **Anti-patterns explicitly rejected:** form grids, 26-field checklists, required-asterisks, "Save" buttons, "Submit" buttons, hard validation blockers, modal stacks.
- **Every visual contract below exists to support one of 13 UI patterns** (streaming ingest, narrative briefing doc, confidence halos, page-wide drop zone, margin notes, discrepancy catcher, soft launch gate, clarification cards, floating chat orb, "Needs Clarity" dashboard tab, in-context hotspots, live pipeline ribbon, auto-save indicator).

**Rule:** No new color constants, no new font stacks, no new global CSS. If a new shade is needed (e.g. a halo fill), it must be expressed as an alpha-composited derivative of an existing token (e.g. `T + '22'`, `R + '1a'`) — following the existing `badgeStyle(color)` pattern.

|-------|-------|-------|

| lg | 24 | Card/section padding, between briefing paragraphs |
| xl | 32 | Layout column gaps, canvas left margin |
| 2xl | 40 | Page horizontal padding (matches `KotoIQShellPage` `padding: '0 40px'`) |
| 3xl | 64 | Briefing doc max-width top/bottom breathing room |

**Exceptions (explicit):**
- Top ribbon (D-23) is **28px tall** post-launch — chosen to match the 36px impersonation-bar pattern already in the shell.

- Floating chat orb is **56×56px** (inherits from `ConversationalBot` launcher).
- Briefing doc body column max-width: **720px** — optimized for prose line-length (65-75 chars at 18px body).

## 3. Color

The shell is 60% white, 30% gray, 10% accent — continuing the existing Koto ratio.

| Role | Value | Token | Usage |
|------|-------|-------|-------|
| Dominant (60%) | `#ffffff` | `W` | Page background, briefing canvas, cards |
| Secondary (30%) | `#f9fafb` | `GRY` | Chat widget scroll bg, margin-note tint, ribbon bg |
| Accent — Koto Pink (≈5%) | `#E6007E` | `R` | Discrepancy catcher dot + callout, primary CTA (Launch button at high-confidence), user message bubble accent |
| Accent — Koto Teal (≈5%) | `#00C2CB` | `T` | Confident-halo fill, citation chips, chat orb, margin-note accent, streaming-narration cursor |
| Success | `#16a34a` | `GRN` | Launch gate at ≥90% confidence, saved indicator, resolved clarification |
| Warning | `#f59e0b` | `AMB` | Launch gate at 70-90% confidence, pale halo (mid-confidence), in-context amber hotspot |
| Text primary | `#111111` | `BLK` | All body text, headings |
| Text muted | `#6b7280` | inline | Subheads, citation source labels, timestamps |
| Divider | `#e5e7eb` | inline | Card borders, section rules, input borders |
| Low-confidence dash border | `#d1d5db` dashed | inline | Dashed outline for fields <0.5 confidence |

**Accent reserved list (explicit — NO interactive element defaults to accent):**
- **Pink (`R`)** — reserved for: (a) the discrepancy catcher (pink dot + pink-outlined callout), (b) the Launch button fill when confidence ≥90%, (c) in-context hotspot dots on fields tied to high-severity clarifications, (d) the destructive confirmation in "Reject field" modal.
- **Teal (`T`)** — reserved for: (a) confident-halo fill for fields ≥0.85 confidence, (b) citation chips, (c) floating chat orb (matches existing `ConversationalBot` launcher), (d) streaming-narration cursor animation, (e) margin-note left border, (f) "saved" success pulse.

**Halo color system (prescriptive):**

| Confidence | Halo style | Color | Notes |
|------------|-----------|-------|-------|
| ≥ 0.85 (confident) | solid box-shadow ring 2px | `T + '33'` (teal 20% alpha) | No visible border; halo is felt, not shouted |
| 0.50 – 0.85 (guessed) | solid box-shadow ring 2px | `AMB + '33'` (amber 20% alpha) | Slightly warmer |
| < 0.50 (missing / low) | 1px dashed border | `#d1d5db` | Plus faint amber background `AMB + '0d'` (5% alpha) |
| Discrepancy (cross-source conflict) | 1px solid pink + pink dot | `R` + pink dot | Overrides confidence halo; discrepancy always wins visually |

**Destructive color:** `#DC2626` (Tailwind red-600) — used only for field-rejection confirmation. This is a NEW color token. Justification: existing theme has no explicit destructive; using Koto Pink for destructive would collide with its accent use. Token name: `DST = '#DC2626'` — add to `src/lib/theme.ts` as part of Phase 7.

## 4. Copywriting Contract

Every user-facing string in Phase 7 is specified below. Executor MUST use these exact strings. The tone is **calm, specific, conversational, never breathless**. Never "wow", "amazing", "fantastic", "absolutely", "certainly" (Alex voice rules apply to text UI too).

### 4.1 Launch Page states

| Element | Copy |
|---------|------|
| Page H1 (before ingest) | `Let's get to know {clientName}.` |
| Page H1 (after ingest) | `{clientName}` (plain, no subtitle — the briefing doc speaks for itself) |
| Ingest prompt (empty state) | `Paste a Koto link, drop a file, or start typing. I'll pull what's already on file and tell you what's missing.` |
| URL input placeholder | `/onboard/:clientId, /onboarding-dashboard/:clientId, or /clients/:clientId` |
| Text-paste textarea placeholder | `Paste a voice transcript, email, meeting notes, or any raw text. I'll extract what matters and cite where each field came from.` |
| Primary CTA (ingest trigger) | `Pull it in` |
| Secondary CTA (add your own) | `+ Add a field` |
| Tertiary CTA (ask your own) | `+ Ask your own question` |

### 4.2 Streaming ingest narration

The narration is Claude speaking. It is NOT a progress bar with percentages. Example template (these are the literal phrases — see D-07, Specifics section of CONTEXT.md):

```
Reading {clientName}'s onboarding... {N} fields found.
Pulling {N} voice calls... analyzing call {i} of {N}...
Found {N} competitor mentions — noted.
Call {i}: operator mentioned "{keyword}" {N} times. Flagging for review.
Pulling discovery doc... {N} sections complete.
Done. Let me show you what I've got.
```

**Voice rules:** lowercase verbs ("pulling", "reading"), ellipses for in-progress steps, em-dash for parenthetical asides, sentence-case.

### 4.3 Briefing doc (narrative prose)

The briefing is typed prose with editable spans. Template (D-06 example is the voice target):

```
{BusinessName} is {a/an} {businessCategory} serving {audienceDescription}
in {serviceArea}. They've been in business since {foundingYear}
{discrepancyNote?}. Their primary service is {primaryService}.
They differentiate on {usps} — {provenanceNote}.
```

**Provenance note style** (lowercase, parenthetical, cited inline):
- `(onboarding form)`
- `(voice call — March 14)`
- `(discovery doc, section 4)`
- `(your note)`

**Discrepancy note style** (inline, flagged):
- `(onboarding said {A}; voice call said "{B}" — flagging)`

## 5. Component-Level Specifications

The 13 UI patterns in the objective, specified one by one.

### 5.1 Streaming ingest narration (D-07)

**Where it lives:** Replaces the ingest-prompt area the moment the operator submits a URL or hits "Pull it in". Fills the 720px briefing column.

**Visual contract:**
- Typography: 16px Raleway, weight 400, line-height 1.55, color `BLK`.
- Each new sentence appears as a new line, fading in (`opacity 0 → 1`, `translateY(6px) → 0`) over 180ms — reuse the `kotoiqBotFadeIn` keyframe already in `ConversationalBot.jsx`.
- Thinking cursor: a 2px × 18px vertical bar in teal (`T`), blinking at 1.2s period (`opacity 0.3 ↔ 1.0`). Sits at the end of the currently-streaming sentence.
- Character-level streaming rate: **cap 40ms per char**, floor 8ms per char (matches `liveFillFields` budget pattern already used in ConversationalBot). Total narration must not exceed 4 seconds regardless of length.
- Completed sentences lose italic styling after they settle (500ms delay). In-progress sentence is italicized.

**States:**
- `idle` — hidden
- `streaming` — narration visible, cursor animating, italic on active sentence
- `settled` — last sentence done, cursor removed, 400ms pause, then narration block fades up and out while the briefing doc fades in
- `error` — shown as a pink-tinted sentence: `I hit a snag pulling {source}. I'll carry on without it.`

**Accessibility:** `aria-live="polite"` on the narration container. `prefers-reduced-motion` MUST disable fades; in reduced-motion, sentences appear instantly with no cursor.

### 5.2 Narrative briefing doc layout (D-06)

**Where it lives:** Main canvas, 720px max-width, horizontally centered with 40px left/right gutters on the shell content area.

**Structure:**
```
[ H1: Client Name ]                                          [ auto-save indicator (top-right) ]
[ Paragraph 1 — identity + category + service area ]
[ Paragraph 2 — primary service + USPs ]
[ Paragraph 3 — audience + differentiators ]
[ Paragraph 4 — discrepancies + flags ]
[ Section rule (1px #e5e7eb) ]
[ H2: "Open questions" ]
[ Ordered list of gaps Claude found — each item is a clarification card inline ]
[ Section rule ]
[ H2: "More I noticed" ]
[ Margin notes inline (D-10) ]
[ 64px breathing room ]
[ Launch gate (sticky-bottom) ]
```

**Editable span contract:**
- Every extracted fact is wrapped in `<span>` with: `padding: 2px 4px; border-radius: 4px; cursor: text; transition: background 120ms;`.
- Halo applied via `box-shadow: 0 0 0 2px {haloColor}` (from §3 halo system). No visible border.
- Hover: `background: T + '0f'` (5% teal).
- Focus (click to edit): expands into an inline `contenteditable` or textarea that retains the halo at 3px ring.
- On blur: auto-saves (see §5.13 auto-save). No Enter/Save button.

**Citation chip contract:**
- Rendered as a small pill immediately after an editable span: `[📎 onboarding]`, `[📞 voice call]`, `[📄 discovery]`, `[✍️ you]`.
- Visual: 10px FH uppercase, weight 700, letter-spacing `.06em`, padding `2px 8px`, radius `20px`, background `T + '15'`, color `T`. (Matches existing `badgeStyle(T)` helper in `src/lib/theme.ts` — do not introduce an override.)
- Hover: expands into a tooltip showing the exact `source_snippet` (≤240 chars), timestamp, and a "jump to source" link.
- Icons: use lucide `FileText` (onboarding), `Phone` (voice), `BookOpen` (discovery), `Edit3` (operator edit). 10px icon.

**Section scroll behavior:** The briefing scrolls naturally inside the shell's content area. No sticky subsections. The launch gate is sticky-bottom (see §5.7).

### 5.3 Confidence halo visual system (D-08)

Already tabled in §3. Implementation notes:

- Halo is implemented as a `box-shadow` (not border) so it does not affect layout width. Pattern: `box-shadow: 0 0 0 2px {color}` for solid, `outline: 1px dashed #d1d5db; outline-offset: 2px` for dashed low-confidence.
- **Radius:** halo inherits the editable span's 4px radius.
- **Never** combine a halo with a visible border — one or the other.
- **Never** animate halo on idle. Halo only animates when: (a) newly populated (pulse once, 600ms, `scale(1.04) → scale(1)`), (b) value changed (same pulse in `GRN`).
- On a discrepancy field (pink), halo is replaced — not layered — by a 1px solid `R` border plus a 6px pink dot positioned `top:-3px; right:-3px`.

### 5.4 Page-wide drop zone (D-09)

**Scope:** The entire `<main>` element of the Launch Page is a drop target. No visible chrome at rest — the affordance is discovered on drag-over.

**States:**
- `resting` — invisible; no border, no message.
- `drag-enter-anywhere-in-window` — dashed 2px teal ring appears 12px inset from viewport edges. Center overlay: 15px FH weight 700 text `Drop it anywhere — I'll figure out what it is.` on a translucent white card (16px radius, 24px padding, `backdrop-filter: blur(4px)`).
- `dropping` (file or URL landed) — ring flashes solid teal once (180ms), then a toast mounts bottom-center reporting outcome (see §4.7 copy).
- `unsupported type in v1` — toast uses amber `AMB` left border: `Got it. I can't parse {fileType} yet — that lands in v2. For now, I'll note it as a source.`

**v1 parser stub:** The drop UI ships fully. v1 handles only: (a) Koto internal URLs → live ingest; (b) raw text paste (via the dedicated textarea, not drop). All other drops create a `kotoiq_client_profile.sources` row with `source_type='deferred_v2'` and trigger the "coming soon" toast.

**Accessibility:** Drop zone is keyboard-reachable via a visible `+ Attach source` button in the top toolbar — the drop zone is a progressive enhancement on top of this button.

### 5.5 Margin notes from Claude (D-10)

**Where they live:** Right margin of the briefing doc, anchored to the sentence they refer to. On viewports < 1120px, they collapse into an inline bubble below the anchor sentence.

**Visual contract (desktop):**
- Container: 220px wide, min-height 64px, background `GRY`, 4px left border in `T`, padding 12px 14px, radius 10px.
- Label row: 11px FH weight 700 uppercase letter-spaced `.06em`, color `T`, icon `Lightbulb` (lucide, 12px) left of label.
- Body: 13px FB weight 500, line-height 1.45, color `BLK`.
- Action row (bottom): three compact buttons in a single row — `Add it` (primary teal, filled), `Skip` (ghost, gray border), `Edit first` (ghost, gray border). Each is 28px tall, 12px font.

**Animation:** Notes slide in from the right (`translateX(20px) → 0`, 240ms ease-out) 800ms after the briefing doc settles. Staggered 120ms per note. Reduced-motion: instant appear.

**Dismissal:** Clicking `Skip` collapses the note upward (height 64px → 0, opacity 1 → 0, 200ms) and leaves a ghost line `Skipped: "{truncated note}"` for 2s before it fully removes.

### 5.6 Discrepancy catcher (D-11)

**The single strongest "wow" moment.** Visually distinct from every other callout.

**Visual contract:**
- Pink dot anchor: 6px diameter, `R`, positioned `top:-3px; right:-3px` on the affected editable span.
- Callout card (expanded on dot click or on paragraph hover): 360px wide, white background, 2px solid `R` border, 14px radius, shadow `0 12px 32px rgba(230,0,126,0.18)`.
- Header row: `AlertTriangle` icon (lucide, 14px, `R`), label `MISMATCH` (11px FH uppercase weight 700 letter-spaced, `R`), spring-loaded close `X` (top-right).
- Body: 14px FB weight 500, line-height 1.5, `BLK`. Renders the conflicting values as inline `<code>`-styled chips (monospace, 13px, `GRY` background, 4px radius) so the reader can scan the conflict in one glance.
- Action row: **one button per source** (primary, teal-filled) + `Edit manually` (ghost) + `Ignore` (text-only link).

**Placement:** Callout renders as a portal, positioned immediately below the paragraph containing the discrepancy. It does NOT float as a tooltip — it's a block element so the reader can interact without losing it on mouse move.

**Post-resolution:** Once operator picks a source, the pink dot animates into a `CheckCircle2` (`GRN`) that fades out over 2s. The chosen value's halo pulses green once.

### 5.7 Soft launch gate readout (D-13, D-14)

**Where it lives:** Sticky-bottom within the briefing canvas. Elevates on scroll (shadow `0 -8px 24px rgba(0,0,0,0.06)`), always visible.

**Layout:**
```
┌─────────────────────────────────────────────────────────────┐
│  [ 94% ]   of what I need to start                          │
│            3 soft gaps remain                               │
│            • service radius specifics                       │
│            • top 3 named competitors                        │
│            • pricing tiers                                  │
│                                                 [  Launch  ]│
└─────────────────────────────────────────────────────────────┘
```

**Visual contract:**
- Height: 120px.
- Left column (percentage): 48px FH weight 900, color tinted by confidence:
  - ≥90%: `GRN`
  - 70-90%: `AMB`
  - <70%: `R`
- Middle column (readout): 15px FB weight 500 for "{N}% of what I need..."; 13px FB weight 600 for "{N} soft gaps remain"; 13px FB weight 400 list items each preceded by a 4px dot in `AMB`.
- Right column (CTA): primary button reusing `buttonPrimary` style but with background tint override:
  - ≥90%: bg `GRN`
  - 70-90%: bg `AMB`, text `#111`
  - <70%: bg `R`
- Button is ALWAYS enabled. Never disabled, never grayed. (D-13 is a soft gate.)
- Left border of the entire readout bar is 4px solid in the confidence color — visible even when scrolled away.

**Post-launch only.** Transitions from the launch gate (§5.7).

### 5.13 Auto-save indicator

**Position:** top-right of the briefing canvas, aligned with the H1.

**Visual contract:**

- `Saved` label fades out over 2 seconds after idle
- No save button. Ever.
- Offline/failure state does NOT block typing — it queues and retries.

**Technical contract (planner note):** Autosave strategy = debounced on 500ms blur OR 2000ms of typing inactivity, matches the existing onboarding form `autosave` pattern. Every save writes to `kotoiq_client_profile` via `/api/kotoiq/profile` route with action `update_field`.

## 9. Accessibility contract

- **Keyboard traversal order:** page H1 → ingest input → paste textarea → page-wide drop zone trigger → briefing editable spans (top-to-bottom document order) → margin note actions → launch gate button → chat orb.
- **Focus ring:** 2px solid `T`, 2px offset, inherits from existing theme. Never removed.
- **ARIA live regions:**
  - Streaming narration container: `aria-live="polite"`, `aria-atomic="false"`.
  - Auto-save indicator: `aria-live="polite"`.
  - Live ribbon: `aria-live="polite"` with `aria-busy` flag during steps.
  - Chat orb badge: `aria-label="{N} pending clarifications"`.
- **Landmarks:** `<main>` wraps the briefing canvas; `<aside>` wraps margin notes and the chat panel; `<nav>` wraps the shell (already present).
- **Color contrast (WCAG AA):** All text ≥ 4.5:1. Teal `T` on white is 3.09:1 — does NOT pass for body text. Rule: **teal text ≥ 14px uses weight 600 AND is only used for labels, chips, and secondary metadata** — never for body prose. Body prose is always `BLK` on white (21:1 ratio).
- **Screen-reader copy for visual-only signals:**
  - Confidence halo: editable span has `aria-describedby="halo-desc-{id}"` pointing to a hidden node reading `Confidence: 72 percent. Sourced from voice call.`
  - Pink discrepancy dot: `aria-label="Conflict between {N} sources. Open to resolve."`
  - Hotspot dot: `aria-label="{N} open questions about this field."`
- **Destructive action:** field-rejection modal has `role="alertdialog"` + focus trap + `Esc` cancels.
